AgenticExplorer — Developer Documentation

Desktop-App zum Verwalten und Überwachen von Claude-CLI-Sessions. Multi-Session-Terminal mit Projekt-Kontext, Kanban, Library und Notizen.

Tauri v2 + React 18 + TypeScript  ·  74 Komponenten · 12 Stores · 9 Hooks · 23 Rust-Module · 47 IPC-Commands

01Überblick

AgenticExplorer ist eine Tauri-v2-Desktop-App. Das Frontend (React 18 + TypeScript + Vite) rendert die UI und hält State in Zustand-Stores. Das Backend (Rust) spawnt PTY-Sessions, liest Dateien sandboxed, und kapselt git / gh. Beide Seiten kommunizieren ausschliesslich über die Tauri-IPC-Bridge — das Frontend ruft invoke(), das Backend pusht emit()-Events.

Multi-Session-Terminal

Mehrere Claude-CLI-Sessions parallel in xterm.js-Terminals, gespawnt als PTYs.

Projekt-Kontext

Config-Panel zeigt CLAUDE.md, Skills, Agents, Hooks, Settings, Worktrees pro Projekt.

Kanban

GitHub Projects v2 Board — Issues per Drag & Drop zwischen Lanes.

Library

Wiederverwendbare Skills/Agents/Hooks/Templates, an Projekte anheftbar.

Session-Diff

Pre-Spawn-Git-Snapshot pro Session — zeigt was die Session geändert hat.

Favoriten & Notizen

Persistierte Projekt-Favoriten, globale + projektbezogene Notizen.

Multi-Window: Neben dem Hauptfenster (main) öffnet die App separate Tauri-Fenster — Log-Viewer, Session-Diff und abgekoppelte Config-Viewer (Commands open_log_window, open_session_diff_window, open_detached_window). Beim Schliessen des main-Fensters werden alle laufenden Sessions sauber beendet. Wird eine einzelne Session geschlossen, emittiert das Backend den Kanal session-deleted/{id} — ein offenes Diff-Fenster friert daraufhin seinen Snapshot ein (DiffWindowView).

02Architektur

Strikte Zwei-Schichten-Trennung: das Frontend kennt keine OS-Primitiven, das Backend kennt kein DOM. Die einzige Brücke ist die typisierte IPC-Schicht.

NOISE_PATTERNS stufen bekanntes Rauschen (Updater-Timeouts, Windows-Ctrl+C-Codes) auf info herab; groupConsecutiveEntries faltet identische Folge-Einträge zu einem Eintrag mit Zähler.

Lesehilfe: Spalte Flow & Methoden nennt die Aufrufkette mit konkreten Funktionen; IPC → Rust den invoke()-Command und das Rust-Modul; Diagramm verweist auf das passende Diagramm in Sektion 6 / 7 bzw. das Komponentendiagramm (2.1). „— (FE)“ = reiner Frontend-Flow ohne Backend-Aufruf.

6.1 Session-Lifecycle (Spawn → Output → Exit)

Laufzeit — Session erstellen und betreiben
sequenceDiagram
  actor User
  participant V as SessionManagerView
  participant H as useSessionEvents
  participant IPC as Tauri IPC
  participant SM as SessionManager (Rust)
  participant PTY as PTY / Shell
  participant Term as xterm.js

  User->>V: "Neue Session" klicken
  V->>IPC: invoke("create_session", {folder, shell, cols, rows})
  IPC->>SM: create_session(...)
  SM->>SM: Pre-Spawn-Git-Snapshot
  SM->>PTY: openpty() + spawn(shell)
  SM->>SM: Reader- + Waiter- + Watcher-Thread starten
  SM-->>IPC: SessionInfo
  IPC-->>V: SessionInfo → sessionStore.addSession()

  loop Terminal-Ausgabe (Reader-Thread)
    PTY-->>SM: Bytes
    SM-->>H: emit("session-output", {id, data})
    H->>Term: term.write(data)
    SM-->>H: emit("session-status", {id, status, snippet})
    Note over SM,H: session-status nur bei Status-Wechsel
(Debounce: ~200/s → ~1-5/s)
    H->>H: sessionStore.updateStatus() + updateLastOutput()
  end

  Note over SM,H: Watcher-Thread: emit("session-claude-id-resolved")
sobald die jsonl-Datei der Session erscheint

  User->>Term: Tastatureingabe
  Term->>IPC: invoke("write_session", {id, data})
  IPC->>PTY: stdin schreiben

  PTY-->>SM: Prozess beendet (Waiter-Thread)
  SM-->>H: emit("session-exit", {id, exit_code})
  H->>H: sessionStore.updateStatus("done")
  Note over SM,H: Bei delete_session: emit("session-deleted/<id>")
(per-session-Channel, friert Diff-Fenster ein — siehe 6.7)
Pro Session laufen drei Threads: Der Reader-Thread liest PTY-Output und emittiert session-output sowie — nur bei Status-Wechsel — session-status (Debounce spart ~99 % der Events). Der Waiter-Thread blockiert auf dem Kindprozess und emittiert session-exit. Der Watcher-Thread beobachtet ~/.claude/projects/ und emittiert session-claude-id-resolved, sobald die jsonl-Datei der Session erscheint — deterministische ID-Zuordnung statt started_at-Heuristik.

6.5 Auto-Updater

Laufzeit — App-Update prüfen, laden, installieren
sequenceDiagram
  actor User
  participant H as useAutoUpdate
  participant UP as plugin-updater
  participant PR as plugin-process
  participant EP as Update-Endpoint
  participant N as UpdateNotification

  Note over H: Mount → 15 s Delay, danach alle 30 min
  H->>UP: Promise.all([check(), getVersion()])
  UP->>EP: latest.json + Signatur abrufen
  UP-->>H: Update? + currentVersion
  H->>H: Vergleich result.version !== currentVersion
(+ dismissedVersionRef gegen Echo)
  alt Neue, nicht-verworfene Version
    H->>N: status "available"
    User->>N: "Herunterladen"
    H->>UP: update.downloadAndInstall(onProgress) — lädt nur
    UP-->>H: Progress-Events → status "downloading"
    UP-->>H: Finished → status "ready"
    User->>N: "Jetzt neu starten" (confirmRelaunch)
    H->>UP: update.install()
    H->>PR: relaunch()
  else User schliesst Notification
    User->>N: dismiss()
    N->>H: dismissedVersionRef.current = version
  else Aktuell oder Netzwerkfehler
    EP-->>H: keine neue Version / Fehler
    H->>H: status "upToDate" → idle
(benigne Netzwerkfehler still verworfen)
  end
Der Updater trennt Download und Installation: download() lädt nur herunter (Status ready) — install() + relaunch() laufen erst nach expliziter Bestätigung. Die Status-Maschine kennt sieben Zustände (idle · checking · available · downloading · ready · upToDate · error); ein verworfenes Update wird pro Version gemerkt und nicht erneut angeboten.

6.8 Cross-Window Preferences-Broadcast

Laufzeit — Einstellung über alle Fenster synchronisieren
sequenceDiagram
  actor User
  participant P as Preferences-Panel
  participant ST as settingsStore
  participant TS as tauriStorage
  participant IPC as Tauri IPC
  participant W2 as Weitere Fenster

  User->>P: Einstellung ändern (Select / Checkbox)
  P->>ST: setPreferences(partial)
  ST->>TS: Persist (settings.json)
  ST->>IPC: emit("preferences-changed", {partial, sourceWindow})
  IPC-->>W2: listen("preferences-changed")
  W2->>W2: Echo-Filter — sourceWindow == eigenes Label?
  alt Fremdes Fenster
    W2->>W2: applyRemotePartial → raw setState

  else Eigenes Fenster
    W2->>W2: ignorieren (Echo-Loop-Schutz)
  end
Jedes Fenster mountet einen eigenen React-Root mit eigenem settingsStore. Der Broadcast hält sie konsistent; der Echo-Filter über sourceWindow verhindert eine Endlosschleife. Ausnahme Theme: theme ist nicht Teil von AppPreferencesSettings — es wird nur über den persistierten Store geteilt, nicht über diesen Broadcast.

6.9 Claude-Session-ID-Auflösung

Laufzeit — UUID einer Session deterministisch zuordnen
sequenceDiagram
  participant WT as Watcher-Thread (Rust)
  participant H as useSessionEvents
  participant IPC as Tauri IPC
  participant FR as file_reader.rs

  Note over WT: Beobachtet ~/.claude/projects/<slug>/
  WT->>WT: Neue jsonl-Datei der Session erscheint
  WT-->>H: emit("session-claude-id-resolved", {id, claude_session_id})
  H->>H: sessionStore.setClaudeSessionId()
  Note over H: Deterministisch — keine Heuristik

  alt Watcher liefert nicht (Fallback nach 3 s, max. 5 Retries)
    H->>IPC: invoke("scan_claude_sessions", {folder})
    IPC->>FR: ~/.claude/projects/* JSONL parsen
    FR-->>H: ClaudeSessionSummary[]

    H->>H: setClaudeSessionId (nur unbeanspruchte UUID)
  end
Der Watcher-Thread ist der bevorzugte, deterministische Pfad — er meldet die UUID, sobald die jsonl-Datei physisch erscheint. Nur falls er ausbleibt, greift die heuristische scan_claude_sessions-Suche mit started_at-Matching. Diese darf eine UUID nicht doppelt vergeben — siehe Lessons-Learned zum Session-Title-Swap-Bug.
